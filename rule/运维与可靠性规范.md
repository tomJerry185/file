## 第一部分：服务 Runbook 规范

### 1.1 Runbook 必须章节

每个核心服务必须包含以下 Runbook 章节：

| # | 章节 | 内容要求 | 规则编号 |
|---|------|----------|----------|
| 1 | 服务概述 | 服务职责、技术栈、团队负责人 | OPS-RB-001 |
| 2 | 健康检查 | 健康检查端点、预期响应、检查脚本 | OPS-RB-002 |
| 3 | 依赖关系 | 上游服务、下游服务、基础设施依赖 | OPS-RB-003 |
| 4 | 关键配置 | 配置文件位置、Nacos 配置项、环境变量 | OPS-RB-004 |
| 5 | 常见故障排查 | 故障现象 → 排查步骤 → 解决方案（至少 5 个场景） | OPS-RB-005 |
| 6 | 扩缩容操作 | 扩容命令、缩容注意事项、HPA 配置 | OPS-RB-006 |
| 7 | 日志与监控 | 日志位置、关键指标、Grafana 面板、告警规则 | OPS-RB-007 |
| 8 | 紧急操作 | 紧急重启、降级开关、数据修复（需审批） | OPS-RB-008 |
| 9 | 变更记录 | 最近变更历史、关联 PR/工单 | OPS-RB-009 |

### 1.2 Runbook 存放位置

MUST：Runbook 与服务代码同目录存放。

```
{service}/docs/runbook.md
```

### 1.3 核心服务 Runbook 模板

#### API Gateway Runbook

**健康检查**：
```
curl -s http://localhost:8080/actuator/health | jq .
# 预期：{"status":"UP"}
```

**依赖关系**：
- 上游：前端（Vue）、移动端
- 下游：user-admin-center, order-center, payment-center, biz-center
- 基础设施：Nacos, Redis, Sentinel

**常见故障**：

| 故障现象 | 可能原因 | 排查步骤 | 解决方案 |
|----------|----------|----------|----------|
| 502 Bad Gateway | 下游服务不可用 | 1. 检查 Nacos 服务列表 2. 检查下游健康 | 启动下游或配置 fallback |
| 429 Too Many Requests | 限流触发 | 1. 检查 Sentinel Dashboard 2. 查看限流规则 | 调整限流阈值 |
| 请求超时 | 下游响应慢 | 1. 查看链路追踪 2. 检查下游日志 | 调整超时或扩容下游 |
| 路由 404 | 路由配置错误 | 1. 检查 Nacos 路由配置 2. 查看 gateway log | 修正路由配置 |

| 网络 QPS | < 5000 | 5000-8000 | > 8000 | 扩容 + CDN 加速 | OPS-RES-007 |
| 线程池活跃率 | < 60% | 60-80% | > 80% | 调整线程池 + 排查阻塞 | OPS-RES-008 |

### 2.3 扩容触发条件

```
自动扩容触发（HPA）：
- CPU > 60% 持续 3 分钟 → 扩容 1 个实例
- CPU > 80% 持续 1 分钟 → 扩容 2 个实例
- 最大实例数：10（非大促）/ 20（大促）

手动扩容条件：
- 大促前 24 小时完成预扩容
- 新版本上线前评估容量
- 新功能上线后观察 7 天
```

## 第三部分：数据库变更安全规范

### 3.1 DDL 变更工具要求

MUST：生产环境 DDL 变更必须使用在线变更工具。

| 工具 | 用途 | 规则编号 |
|------|------|----------|
| pt-online-schema-change | MySQL 大表 DDL | OPS-DB-001 |
| gh-ost | MySQL 大表 DDL（GitHub 开源） | OPS-DB-002 |
| 小表直接 ALTER | 行数 < 10 万且非高峰期 | OPS-DB-003 |

### 3.2 大表操作要求

MUST：大表（> 100 万行）操作必须遵循以下规范。

```
加字段：
- 使用 pt-online-schema-change 或 gh-ost
- 禁止添加 NOT NULL 字段（除非有默认值）
- 禁止使用 ENUM 类型变更

加索引：
- 使用 pt-online-schema-change 在线创建
- 禁止在生产高峰期创建
- 预估索引大小（避免磁盘空间不足）

修改字段类型：
- 先添加新字段 → 迁移数据 → 删除旧字段
- 禁止直接 ALTER COLUMN
```

### 3.3 数据迁移规范

MUST：大批量数据迁移必须分批执行。

```
分批迁移 SOP：
1. 编写迁移脚本（支持断点续传）
2. 在测试环境验证（数据量 = 生产 1/10）
3. 申请维护窗口
4. 备份生产数据
5. 分批执行（每批 < 1 万行，间隔 5 秒）
6. 每批后校验数据一致性
7. 全部完成后清理临时字段/表
8. 更新应用代码引用
```

### 3.4 变更审批流程

| 风险等级 | 审批人 | 变更窗口 |
|----------|--------|----------|
| 低风险（添加索引、小表 ALTER） | Tech Lead | 任意非高峰 |
| 中风险（大表加字段、修改字段） | Tech Lead + DBA | 凌晨 02:00-05:00 |
| 高风险（删除表/字段、修改主键） | DBA + 架构师 | 维护窗口 |
| 数据修正（UPDATE/DELETE 大量数据） | Tech Lead + DBA | 凌晨 02:00-05:00 |

### 3.5 变更窗口

MUST：高风险变更只能在低峰期执行。

```
变更窗口：02:00 - 05:00（北京时间）
禁止变更窗口：10:00 - 22:00（业务高峰期）
紧急变更：SRE 值班批准 + 事后补充审批
```

### 3.6 变更后验证

| 验证项 | 方法 | 规则编号 |
|--------|------|----------|
| 数据一致性 | 抽样校验 + COUNT 对比 | OPS-DB-004 |
| 性能对比 | 变更前后 EXPLAIN 对比 | OPS-DB-005 |
| 应用功能 | 冒烟测试 | OPS-DB-006 |
| 回滚方案 | 验证回滚脚本可用 | OPS-DB-007 |

## 第四部分：灾备与业务连续性规范

### 4.1 RTO/RPO 目标

| 服务 | RTO | RPO | 优先级 |
|------|-----|-----|--------|
| API Gateway | < 5 分钟 | < 1 分钟 | P0 |
| user-admin-center | < 30 分钟 | < 5 分钟 | P1 |
| order-center | < 30 分钟 | < 5 分钟 | P1 |
| payment-center | < 30 分钟 | < 5 分钟 | P0 |
| biz-center | < 1 小时 | < 30 分钟 | P1 |
| message-center | < 2 小时 | < 1 小时 | P2 |
| 其他服务 | < 4 小时 | < 1 小时 | P2 |

### 4.2 核心交易链路降级预案

| 故障场景 | 降级方案 | 自动/手动 | 触发条件 |
|----------|----------|-----------|----------|
| 支付服务不可用 | 切换到货到付款模式 | 手动 | 支付成功率 < 50% |
| 库存服务不可用 | 人工核库存 + 限流 | 手动 | 库存查询超时 > 5s |
| 短信服务不可用 | 邮件验证码 + APP 推送 | 自动 | 短信发送失败 > 3 次 |
| 搜索服务不可用 | 数据库直接查询（限流） | 自动 | 搜索超时率 > 30% |
| 推荐服务不可用 | 返回热门商品列表 | 自动 | 推荐超时 > 2s |
| AI 服务不可用 | 返回预设回复 | 自动 | AI 响应超时 > 10s |

### 4.3 中间件故障应对

| 中间件 | 故障应对 | 数据安全 | 恢复方式 |
|--------|----------|----------|----------|
| Redis 不可用 | 直接查 DB + 限流降级 | DB 有完整数据 | Redis 恢复后自动预热 |
| MySQL 不可用 | 只读副本接管 | 主从同步可能有延迟 | 主库恢复后切回 |
| RocketMQ 不可用 | 本地消息表暂存 | 消息不丢失 | MQ 恢复后补偿发送 |
| Nacos 不可用 | 本地配置缓存 | 配置可能有延迟 | Nacos 恢复后自动同步 |
| Sentinel 不可用 | 降级为本地规则 | 限流规则可能不更新 | Sentinel 恢复后自动同步 |

### 4.4 故障演练计划

MUST：每季度进行一次故障演练。

| 季度 | 演练场景 | 验收标准 |
|------|----------|----------|
| Q1 | 单服务宕机 | 自动恢复 + 告警正常 + 业务不受影响 |
| Q2 | Redis 主节点宕机 | Sentinel 自动切换 + 应用无感知 |
| Q3 | MySQL 主库宕机 | 从库接管 + 数据一致性校验 |
| Q4 | 全链路压测 | 所有服务不崩溃 + 响应时间可接受 |

### 4.5 数据一致性恢复 SOP

分布式事务失败后的数据修复步骤：

```
1. 确认事务状态（查询 outbox_event 表）
2. 识别不一致的数据（对比各服务状态）
3. 制定修复方案：
   a. 补偿操作（如退款、取消订单）
   b. 重试操作（如重新发送消息）
   c. 人工修正（如手动对账）
4. 在测试环境验证修复脚本
5. 在生产环境执行（先小批量验证）
6. 验证数据一致性
7. 记录修复过程和审计日志
```

## 第五部分：高阶灾备方案

> [蓝图] 本章为长期演进目标，当前系统处于单机房主从阶段，以下方案为分阶段演进路线。

### 5.1 灾备架构演进路线

```
当前状态          阶段一           阶段二           阶段三
单机房主从  →  同城双活  →  两地三中心  →  异地多活 + 单元化
   │              │               │                │
   RPO:分钟级     RPO:秒级        RPO:秒级         RPO:0（无损）
   RTO:小时级     RTO:分钟级      RTO:分钟级       RTO:秒级
```

### 5.2 同城双活方案

> 目标：单机房故障时业务不中断，RPO < 30 秒，RTO < 1 分钟。

#### 5.2.1 架构拓扑

```
                    DNS/GSLB
                   /        \
           机房 A（主）    机房 B（备）
           =========      =========
           Nacos 集群     Nacos 集群（跨机房同步）
           Redis Sentinel Redis Sentinel（跨机房主从）
           MySQL 主       MySQL 从 → 提升为主
           RocketMQ      RocketMQ
           App 实例群     App 实例群
```

#### 5.2.2 关键组件规范

| 组件 | 同城双活方案 | 数据同步 | 规则编号 |
|------|-------------|----------|----------|
| MySQL | 主从半同步复制 + GTID | 异步/半同步，延迟 < 1s | OPS-DR-001 |
| Redis | Redis Sentinel 跨机房部署 | 异步复制，切换时丢少量数据 | OPS-DR-002 |
| RocketMQ | 同步双写 + 异步刷盘 | 消息双写到两个机房 Broker | OPS-DR-003 |
| Nacos | Nacos 集群跨机房部署 | 集群内自动同步 | OPS-DR-004 |
| 应用层 | 两个机房各部署完整服务 | 无状态服务多机房部署 | OPS-DR-005 |

#### 5.2.3 流量调度

MUST：同城双活流量调度使用 DNS + 负载均衡双层调度。

```
流量调度策略：
- 正常：DNS 按权重分配（机房 A 70% / 机房 B 30%）
- 机房 A 故障：DNS 自动切换到机房 B 100%
- 切换方式：GSLB 健康检查 + 自动 failover
- 切换时间：< 30 秒（DNS TTL 设为 30 秒）
```

#### 5.2.4 数据一致性保障

| 场景 | 一致性要求 | 保障方案 | 规则编号 |
|------|-----------|----------|----------|
| 订单创建 | 强一致 | 同机房 MySQL 主库写入，异步同步到备机房 | OPS-DR-006 |
| 库存扣减 | 强一致 | Redis 分布式锁 + Lua 脚本原子操作 | OPS-DR-007 |
| 消息投递 | 至少一次 | RocketMQ 同步双写 + 消费幂等 | OPS-DR-008 |
| 配置变更 | 最终一致 | Nacos 集群自动同步（秒级） | OPS-DR-009 |

### 5.3 异地多活方案

> 目标：城市级灾难下业务可恢复，RPO < 30 秒，RTO < 3 分钟。

#### 5.3.1 两地三中心架构

```
          全局 GSLB
         /    |     \
    北京同城    上海远端
    ┌────────┐  ┌────────┐
    │ 机房 A │  │ 机房 C │
    │(主)    │  │(灾备)  │
    │        │  │        │
    │ 机房 B │  │        │
    │(同城备)│  │        │
    └────────┘  └────────┘
```

#### 5.3.2 核心挑战与对策

| 挑战 | 解决方案 | 规则编号 |
|------|----------|----------|
| 跨城延迟（20-50ms） | 本地读 + 异步写，核心链路同城闭环 | OPS-MULTI-001 |
| 数据冲突 | 按用户 ID 分片（单元化），避免跨机房写 | OPS-MULTI-002 |
| 全局唯一 ID | 雪花算法（数据中心位区分机房） | OPS-MULTI-003 |
| 配置同步 | Nacos 跨城集群 + 本地缓存兜底 | OPS-MULTI-004 |
| 灰度发布 | 按机房逐步放量 | OPS-MULTI-005 |

#### 5.3.3 异地多活分级策略

| 服务等级 | 多活策略 | 示例 |
|----------|----------|------|
| 核心交易 | 单元化多活（按用户分片） | 订单、支付、账户 |
| 重要服务 | 同城双活 + 异地灾备 | 商品、用户、消息 |
| 一般服务 | 主从灾备 | 营销、AI、报表 |

### 5.4 单元化架构

> [蓝图] 单元化是实现异地多活的核心技术手段，将系统按用户维度划分为独立的"单元"。

#### 5.4.1 单元化模型

```
用户请求 → GSLB → 单元路由（按 user_id hash）
                      │
              ┌───────┼───────┐
              ▼       ▼       ▼
           单元 A   单元 B   单元 C
           ┌────┐  ┌────┐  ┌────┐
           │App │  │App │  │App │
           │DB  │  │DB  │  │DB  │
           │Redis│  │Redis│  │Redis│
           │MQ  │  │MQ  │  │MQ  │
           └────┘  └────┘  └────┘
              │       │       │
              └───────┼───────┘
                      ▼
                  全局服务
              （配置/搜索/AI）
```

#### 5.4.2 单元化核心规范

| 规则 | 说明 | 规则编号 |
|------|------|----------|
| 用户维度分片 | 按 user_id hash 到固定单元，用户所有请求在同一单元闭环 | OPS-UNIT-001 |

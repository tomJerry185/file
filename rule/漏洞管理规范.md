## 1. 扫描策略与频率

> MUST：安全扫描覆盖所有层面，频率满足最低要求。

### 1.1 扫描矩阵

| 扫描类型 | 扫描对象 | 频率 | 工具 | 触发方式 | 规则编号 |
|----------|----------|------|------|----------|----------|
| SAST | 源代码 | 每次 PR | Semgrep / SonarQube | CI 自动 | SEC-SCAN-001 |
| DAST | 运行中的应用 | 每周 | OWASP ZAP | 定时任务 | SEC-SCAN-002 |
| 依赖扫描 | Maven/Node 依赖 | 每次构建 | OWASP Dependency-Check | CI 自动 | SEC-SCAN-003 |
| 镜像扫描 | Docker 镜像 | 每次推送 | Trivy | CI 自动 | SEC-SCAN-004 |
| 密钥扫描 | 代码仓库 | 每次 PR | GitLeaks / TruffleHog | CI 自动 | SEC-SCAN-005 |
| IaC 扫描 | K8s/TF 配置 | 每次 PR | Checkov / tfsec | CI 自动 | SEC-SCAN-006 |
| 许可证合规 | 依赖许可证 | 每周 | FOSSA / license-checker | 定时任务 | SEC-SCAN-007 |

## 2. 漏洞处理 SLA

> MUST：漏洞修复时限与业务模块风险等级挂钩。

### 2.1 通用 SLA

| 严重程度 | CVSS 范围 | 通用 SLA | 规则编号 |
|----------|----------|----------|----------|
| Critical | ≥ 9.0 | 7 天 | SEC-SLA-001 |
| High | 7.0 - 8.9 | 30 天 | SEC-SLA-002 |
| Medium | 4.0 - 6.9 | 90 天 | SEC-SLA-003 |
| Low | < 4.0 | 180 天 | SEC-SLA-004 |

### 2.2 业务模块加权 SLA

> 核心模块的 Critical 漏洞需更快速修复。

| 业务模块 | Critical | High | Medium | 说明 |
|----------|----------|------|--------|------|
| 支付中心 | **24 小时** | 7 天 | 30 天 | 资金安全 |
| 用户中心 | **48 小时** | 14 天 | 60 天 | 身份安全 |
| 订单中心 | **3 天** | 14 天 | 60 天 | 交易安全 |
| API 网关 | **48 小时** | 7 天 | 30 天 | 全局入口 |
| 其他模块 | 7 天 | 30 天 | 90 天 | 通用 SLA |

### 2.3 SLA 计时规则

- 开始时间：漏洞被确认（非发现时间）
- 暂停条件：等待第三方修复（如开源库未发布补丁）
- 超时升级：SLA 超时 50% → 升级至 Tech Lead，SLA 超时 → 升级至 CTO

## 3. 临时缓解措施

> MUST：无法立即修复的漏洞必须采取临时缓解措施。

### 3.1 缓解措施清单

| 措施 | 适用场景 | 有效性 | 规则编号 |
|------|----------|--------|----------|
| 版本锁定 | 依赖有漏洞但无补丁 | 低 | SEC-MIT-001 |
| WAF 规则 | Web 应用漏洞 | 中 | SEC-MIT-002 |
| 网络隔离 | 基础设施漏洞 | 高 | SEC-MIT-003 |
| 功能禁用 | 漏洞功能非核心 | 高 | SEC-MIT-004 |
| 增强监控 | 已知漏洞但风险可控 | 低 | SEC-MIT-005 |
| 降级方案 | 核心模块不可用 | 高 | SEC-MIT-006 |

### 3.2 缓解措施 SOP

```
1. 评估漏洞可利用性（是否有公开 PoC）
2. 评估业务影响（漏洞模块的业务价值）
3. 选择缓解措施
4. 实施缓解措施
5. 增强监控（针对该漏洞的异常告警）
6. 跟踪补丁发布
7. 补丁发布后实施永久修复
8. 移除临时缓解措施
```

## 4. 漏洞信息源

> SHOULD：主动订阅安全信息源，提前发现潜在漏洞。

| 信息源 | 类型 | 订阅方式 | 用途 |
|--------|------|----------|------|
| NVD (nvd.nist.gov) | 通用漏洞库 | RSS / API | 依赖漏洞查询 |
| GitHub Advisory | 开源依赖 | 自动（Dependabot） | 依赖安全告警 |
| Spring Security Advisories | 框架安全 | RSS | Spring 漏洞跟踪 |
| Alibaba Cloud Security | 云安全 | 控制台 | 云平台漏洞 |
| CNVD (cnvd.org.cn) | 国内漏洞库 | 邮件订阅 | 国内安全动态 |
| OSS Security | 开源项目安全 | GitHub Watch | 直接依赖的安全公告 |

## 5. 新依赖安全评估

> MUST：引入新依赖前必须进行安全评估。

### 5.1 评估检查清单

| 检查项 | 要求 | 规则编号 |
|--------|------|----------|
| 已知漏洞 | 无 Critical/High 漏洞 | SEC-DEP-001 |
| 维护状态 | 最近 6 个月有更新 | SEC-DEP-002 |
| 许可证 | 兼容项目许可证（Apache 2.0 / MIT） | SEC-DEP-003 |
| 下载量 | Maven Central / npm 下载量 > 10k/月 | SEC-DEP-004 |
| 社区活跃度 | Star > 100，Issue 响应 < 7 天 | SEC-DEP-005 |
| 传递依赖 | 检查传递依赖中的漏洞 | SEC-DEP-006 |
| 替代方案 | 是否有更活跃的替代库 | SEC-DEP-007 |

### 5.2 评估流程

```
1. 研发提交依赖引入申请（PR）
2. CI 自动运行依赖安全扫描
3. 安全团队审核（High 风险依赖需人工审核）
4. Tech Lead 批准
5. 合并后进入 CI 常规扫描
```

## 6. 容器镜像安全标准

> MUST：所有生产容器镜像必须经过安全扫描。

### 6.1 基础镜像要求

| 要求 | 标准 | 规则编号 |
|------|------|----------|
| 基础镜像 | 推荐使用 Alpine / Distroless | SEC-IMG-001 |
| 镜像大小 | < 200 MB（应用镜像） | SEC-IMG-002 |
| 用户权限 | 非 root 用户运行 | SEC-IMG-003 |
| 可执行文件 | 仅包含必要二进制 | SEC-IMG-004 |
| Shell | 生产镜像不包含 shell（Distroless） | SEC-IMG-005 |

### 6.2 镜像安全检查

| 检查项 | 工具 | 频率 |
|--------|------|------|
| 已知 CVE | Trivy | 每次构建 |
| 镜像签名 | Cosign / Notary | 每次推送 |
| 基础镜像版本 | Trivy | 每周 |
| 敏感文件检测 | Trivy / Docker scout | 每次构建 |

### 6.3 禁止事项

MUST：生产镜像中禁止包含：
- 密码、密钥、Token
- .git 目录
- 调试工具（curl、wget、vim）— 除非明确需要
- 开发依赖

## 7. SBOM 管理

> SHOULD：自动生成和维护软件物料清单（SBOM）。

### 7.1 SBOM 生成

| 生成时机 | 工具 | 格式 | 存储位置 |
|----------|------|------|----------|
| Maven 构建 | CycloneDX Maven Plugin | JSON | 仓库 /artifact/ |
| Docker 构建 | Syft | JSON | 镜像标签附加 |
| npm 构建 | CycloneDX npm Plugin | JSON | 仓库 /artifact/ |

### 7.2 SBOM 使用

- 依赖漏洞关联：快速定位受影响的服务
- 许可证合规：自动检查许可证冲突
- 供应链安全：依赖变更审计

## 8. CI 自动化集成

> MUST：安全扫描集成到 CI/CD 流水线，阻断高危漏洞。

### 8.1 Maven 集成 OWASP Dependency-Check

已在根 pom.xml 添加 maven-checkstyle-plugin，建议补充：

```xml
<!-- OWASP Dependency-Check（建议在根 pom.xml 添加） -->
<plugin>
    <groupId>org.owasp</groupId>
    <artifactId>dependency-check-maven</artifactId>
    <version>9.0.8</version>
    <configuration>
        <failBuildOnCVSS>7</failBuildOnCVSS>
        <suppressionFile>rule/automation/dependency-check-suppressions.xml</suppressionFile>
    </configuration>
    <executions>
        <execution>
            <goals>
                <goal>check</goal>
            </goals>
        </execution>
    </executions>
</plugin>
```

### 8.2 CI Workflow 扫描

建议在 CI 中添加以下步骤：

```yaml
# .github/workflows/security-scan.yml
security-scan:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Trivy filesystem scan
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        severity: 'CRITICAL,HIGH'
        exit-code: 1
    - name: Gitleaks secret scan
      uses: gitleaks/gitleaks-action@v2
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

## 9. 度量与报告

### 9.1 关键指标

| 指标 | 目标 | 计算方式 |
|------|------|----------|
| 漏洞修复率（按 SLA） | > 95% | 按时修复数 / 总漏洞数 |
| Critical 漏洞平均修复时间 | < 3 天 | 从确认到修复的平均天数 |
| 开放漏洞数量趋势 | 持续下降 | 按月统计开放漏洞数 |
| 扫描覆盖率 | 100% | 被扫描的服务数 / 总服务数 |
| 新引入漏洞数 | 0 | 每次构建新增漏洞数 |

### 9.2 报告频率

| 报告类型 | 频率 | 接收人 |
|----------|------|--------|
| 漏洞周报 | 每周 | 安全团队 + Tech Lead |
| 安全月报 | 每月 | 全体技术团队 + 管理层 |
| 季度安全审计 | 每季度 | CTO + 审计委员会 |

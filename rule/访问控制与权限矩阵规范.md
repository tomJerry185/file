## 1. 生产环境访问控制

> MUST：生产环境访问遵循最小权限原则。

### 1.1 SSH 访问控制

| 角色 | 权限范围 | 认证方式 | 审批流程 |
|------|----------|----------|----------|
| SRE | 所有生产服务器 | SSH Key + MFA | Tech Lead 审批 |
| 后端开发 | 指定服务容器 | kubectl exec（只读） | SRE 审批 |
| DBA | 数据库服务器 | SSH Key + MFA +堡垒机 | CTO 审批 |
| 外部供应商 | 无（禁止直接访问） | VPN + 堡垒机 + 时间窗口 | CTO + 法务审批 |

### 1.2 Break-Glass 紧急访问

紧急情况下（P0/P1 事故），允许临时提权：

```text
1. 申请人发起 Break-Glass 请求（钉钉/飞书）
2. 值班 SRE 确认事故等级
3. 临时授权：
   - 有效期：最长 2 小时
   - 范围：仅事故相关服务
   - 操作记录：全程录屏/命令审计
4. 事故结束后立即收回权限
5. 48 小时内补充审批流程
6. 事后审计：审查所有临时操作
```

### 1.3 堡垒机规则

MUST：所有生产服务器访问必须经过堡垒机。

MUST：堡垒机启用命令审计和会话录像。

SHOULD：高危命令（rm -rf、DROP、DELETE WHERE 1=1）需二次确认。

## 2. 数据库权限分离

> MUST：数据库账号按职责分离，禁止使用超级账号。

### 2.1 账号体系

| 账号类型 | 权限 | 用途 | 管理方式 |
|----------|------|------|----------|
| 应用账号 | DML（SELECT/INSERT/UPDATE/DELETE） | 应用运行时访问 | 自动化配置管理 |
| 只读账号 | SELECT | 报表/监控/客服查询 | DBA 管理 |
| DDL 账号 | DDL（CREATE/ALTER/DROP） | 数据库结构变更 | DBA 管理 + 审批 |
| 管理账号 | ALL | 数据库运维 | DBA + CTO 双人授权 |
| 审计账号 | SELECT（审计表） | 安全审计 | 安全团队管理 |

### 2.2 权限规范

MUST：应用账号禁止有 DDL 权限。

MUST：应用账号禁止跨库访问。

MUST：每个微服务使用独立数据库账号。

```sql
-- 应用账号示例
CREATE USER 'app_order'@'%' IDENTIFIED BY '${DB_PASSWORD}';
GRANT SELECT, INSERT, UPDATE, DELETE ON shop_order.* TO 'app_order'@'%';
FLUSH PRIVILEGES;

-- 禁止
-- GRANT ALL ON *.* TO 'app_order'@'%';  -- 绝对禁止
-- GRANT DROP, ALTER ON shop_order.* TO 'app_order'@'%';  -- 应用不需要
```

## 3. 服务间调用鉴权

> MUST：内部微服务间 API 调用必须经过认证。

### 3.1 认证方式

| 场景 | 认证方式 | 说明 |
|------|----------|------|
| API Gateway → 后端服务 | JWT Token | Gateway 签发，后端验证 |
| 后端服务 → 后端服务 | Internal API Key 或 mTLS | 服务间信任 |
| 前端 → API Gateway | Sa-Token Session / JWT | 用户身份认证 |
| 第三方 → API Gateway | API Key + IP 白名单 | 外部合作方 |

### 3.2 内部 API Key 规范

MUST：服务间调用使用预共享密钥（API Key）。

```text
Header: X-Internal-Token: ${SERVICE_TOKEN}
```

MUST：API Key 通过环境变量注入，禁止硬编码。

MUST：API Key 定期轮换（至少每季度一次）。

### 3.3 mTLS 规范（推荐）

SHOULD：核心链路（支付、订单）使用双向 TLS 认证。

```text
mTLS 配置：
- 证书颁发：内部 CA（HashiCorp Vault / cert-manager）
- 证书有效期：90 天
- 自动续期：cert-manager + Kubernetes
- 吊销机制：CRL / OCSP
```

## 4. RBAC 权限矩阵

> MUST：所有用户操作必须有明确的角色权限定义。

### 4.1 角色定义

| 角色编码 | 角色名称 | 说明 |
|----------|----------|------|
| SUPER_ADMIN | 超级管理员 | 系统最高权限，仅 CTO/架构师 |
| ADMIN | 管理员 | 业务管理员，管理商品/订单/用户 |
| MERCHANT | 商家 | 管理自己的商品和订单 |
| CUSTOMER | 普通用户 | 浏览、下单、支付、评价 |
| CUSTOMER_SERVICE | 客服 | 查看订单、处理售后 |
| OPERATOR | 运营 | 管理活动、促销、内容 |
| FINANCE | 财务 | 查看财务报表、退款审核 |
| SRE | 运维 | 查看监控、管理配置 |

### 4.2 权限矩阵（角色 × 资源 × 操作）

| 资源 | 操作 | SUPER_ADMIN | ADMIN | MERCHANT | CUSTOMER | CS | OPERATOR | FINANCE | SRE |
|------|------|-------------|-------|----------|----------|----|----------|---------|-----|
| 用户管理 | 查看 | R | R | - | O(自己) | R | R | - | - |
| 用户管理 | 编辑/删除 | R/W/D | R/W | - | - | - | - | - | - |
| 商品管理 | 查看 | R | R | O(自己) | R | R | R | - | - |
| 商品管理 | 创建/编辑 | R/W | R/W | O(自己) | - | - | R/W | - | - |
| 商品管理 | 上下架 | R/W | R/W | O(自己) | - | - | R/W | - | - |
| 订单管理 | 查看 | R | R | O(自己) | O(自己) | R | R | R | - |
| 订单管理 | 修改状态 | R/W | R/W | O(自己) | - | - | - | - | - |
| 订单管理 | 退款审核 | R/W | R/W | - | - | - | - | R/W | - |
| 支付管理 | 查看 | R | R | O(自己) | O(自己) | R | - | R | - |
| 系统配置 | 查看 | R | - | - | - | - | - | - | R |
| 系统配置 | 修改 | R/W | - | - | - | - | - | - | R(限运维) |
| 监控面板 | 查看 | R | - | - | - | - | - | - | R |
| 数据导出 | 导出 | R | R(审批) | - | - | - | R(审批) | R | - |
| 审计日志 | 查看 | R | R | - | - | - | - | R | R |

> R = Read, W = Write, D = Delete, O = Own（自己的数据）

### 4.3 权限实现规范

MUST：使用注解控制接口权限。

```java
// Sa-Token 权限注解
@SaCheckRole("ADMIN")           // 角色检查
@SaCheckPermission("order:edit") // 权限检查

// 数据级别权限
public List<Order> getOrders() {
    // 根据当前用户角色过滤数据
    if (StpUtil.hasRole("ADMIN")) {
        return orderMapper.selectList(null); // 全部
    } else if (StpUtil.hasRole("MERCHANT")) {
        return orderMapper.selectByMerchantId(currentMerchantId()); // 自己的
    } else {
        return orderMapper.selectByUserId(currentUserId()); // 自己的
    }
}
```

## 5. 特权操作审计

> MUST：所有 DDL、生产数据修改、权限变更必须记录审计日志。

### 5.1 审计事件列表

| 事件类型 | 审计内容 | 保留期 |
|----------|----------|--------|
| DDL 变更 | 操作人、SQL、影响行数、执行时间 | 永久 |
| 数据修改 | 操作人、SQL（脱敏）、影响行数 | 永久 |
| 权限变更 | 操作人、变更内容、变更前后 | 永久 |
| 配置变更 | 操作人、配置项、变更前后 | 1 年 |
| 用户登录/登出 | 用户ID、IP、时间、结果 | 90 天 |
| 敏感数据访问 | 访问人、数据类型、访问时间 | 90 天 |
| Break-Glass | 申请人、授权人、操作内容、时长 | 永久 |

### 5.2 审计日志格式

```json
{
  "timestamp": "2026-03-31T10:00:00.000+08:00",
  "eventType": "DDL_CHANGE",
  "operator": "admin_zhangsan",
  "operatorIp": "10.0.0.100",
  "resource": "shop_order.orders",
  "action": "ALTER TABLE",
  "detail": "ADD COLUMN refund_status VARCHAR(32)",
  "result": "SUCCESS",
  "duration": 1523,
  "traceId": "abc123def456"
}
```

### 5.3 审计日志存储

MUST：审计日志写入独立的审计库，禁止业务系统修改或删除。

SHOULD：审计日志同步到 ELK 或 Loki 进行检索。

## 7. ABAC 属性基访问控制规范

> MUST：多租户、细粒度权限场景必须采用 ABAC 模型补充 RBAC 的不足。

### 7.1 RBAC + ABAC 混合模型

本项目采用 RBAC（角色）+ ABAC（属性）混合权限模型：

| 模型 | 适用场景 | 示例 |
|------|----------|------|
| RBAC | 粗粒度功能权限 | ADMIN 可管理商品、MERCHANT 可编辑自己的商品 |
| ABAC | 细粒度数据权限 | 商家只能查看自己店铺的订单、客服只能处理自己负责区域的工单 |

```text
权限判断流程：
请求 → 认证（Token 验证）→ RBAC 检查（角色权限）→ ABAC 检查（属性策略）→ 放行/拒绝
                                                ↑
                                    环境属性 + 资源属性 + 主体属性
```

### 7.2 ABAC 策略要素（PEP + PDP + PIP）

| 要素 | 全称 | 职责 | 实现建议 |
|------|------|------|----------|
| PEP | Policy Enforcement Point | 拦截请求、执行决策 | 自定义注解 + AOP |
| PDP | Policy Decision Point | 评估策略、返回决策 | PermissionService |
| PIP | Policy Information Point | 提供属性数据 | 上下文/数据库/缓存 |

### 7.3 多租户隔离规范

> MUST：多租户场景必须实现数据隔离。

| 隔离级别 | 实现方式 | 适用场景 | 规则编号 |
|----------|----------|----------|----------|
| 逻辑隔离 | tenant\_id 字段过滤 | SaaS 平台、商家入驻 | SEC-ABAC-001 |
| Schema 隔离 | 每租户独立 Schema | 数据敏感型租户 | SEC-ABAC-002 |
| 物理隔离 | 每租户独立数据库实例 | 金融、医疗等强合规场景 | SEC-ABAC-003 |

MUST：所有涉及租户数据的查询必须自动注入 `tenant_id` 条件，禁止通过 API 参数传递。

```java
// ABAC 数据隔离拦截器
@Aspect
@Component
public class TenantDataIsolationAspect {

    @Before("@annotation(tenantRequired)")
    public void enforceTenantIsolation(JoinPoint joinPoint, TenantRequired tenantRequired) {
        Long tenantId = TenantContext.getCurrentTenantId();
        if (tenantId == null) {
            throw new AccessDeniedException("缺少租户上下文");
        }
        // 自动追加 WHERE tenant_id = ? 条件
        TenantContext.setRequired(true);
    }
}
```

### 7.4 细粒度权限策略定义

> MUST：关键资源操作必须定义 ABAC 策略。

| 策略类型 | 属性组合 | 示例 | 规则编号 |
|----------|----------|------|----------|
| 行级权限 | 主体 ID + 资源 owner\_id | 用户只能编辑自己的地址 | SEC-ABAC-004 |
| 列级权限 | 角色 + 数据分级 | 客服看不到银行卡完整号 | SEC-ABAC-005 |
| 时间权限 | 角色 + 时间窗口 | 运营活动在指定时间段可编辑 | SEC-ABAC-006 |
| IP 权限 | IP 段 + 操作类型 | 管理后台仅允许办公网 IP | SEC-ABAC-007 |
| 数量权限 | 角色 + 操作次数 | 普通用户每日退款不超过 3 次 | SEC-ABAC-008 |

### 7.5 ABAC 实现规范

```java
// 细粒度权限注解
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AbacCheck {
    String resource();   // 资源类型：order, product, user
    String action();     // 操作类型：read, write, delete
    String condition() default "";  // SpEL 条件表达式
}

// 使用示例
@AbacCheck(resource = "order", action = "write",
           condition = "#order.merchantId == authentication.principal.merchantId")
public void updateOrder(@RequestBody Order order) {
    // 仅当订单属于当前商家时允许修改
}
```

MUST：ABAC 策略变更必须记录审计日志。

SHOULD：ABAC 策略支持动态配置（Nacos / 数据库），避免硬编码。

### 7.6 多租户资源配额

| 租户类型 | 商品上限 | 存储配额 | API QPS | 规则编号 |
|----------|----------|----------|---------|----------|
| 免费租户 | 100 | 1 GB | 50/分钟 | SEC-ABAC-009 |
| 基础租户 | 1,000 | 10 GB | 200/分钟 | SEC-ABAC-010 |
| 高级租户 | 10,000 | 100 GB | 1,000/分钟 | SEC-ABAC-011 |
| 企业租户 | 不限 | 不限 | 自定义 | SEC-ABAC-012 |

MUST：配额检查在网关层拦截，避免请求到达业务服务。

## 第一部分：微服务集成与契约测试规范

### 1.1 服务间接口契约定义

MUST：所有服务间接口必须使用 OpenAPI 3.0 规范定义。

| 规则 | 级别 | 规则编号 |
|------|------|----------|
| API 接口必须提供 OpenAPI 3.0 定义文件 | MUST | TEST-CON-001 |
| 接口变更必须向后兼容（新增可选字段允许） | MUST | TEST-CON-002 |
| 破坏性变更必须有版本号升级（v1 → v2） | MUST | TEST-CON-003 |
| 废弃接口必须有 @Deprecated 标记和迁移文档 | MUST | TEST-CON-004 |
| OpenAPI 文件与代码同步更新 | MUST | TEST-CON-005 |

### 1.2 契约测试要求

MUST：消费方驱动的契约测试（CDC），服务提供方验证契约。

```
契约测试策略：
- 消费方（Consumer）编写期望的 API 响应契约
- 提供方（Provider）在 CI 中验证是否能满足所有契约
- 契约文件存放在 consumer/docs/contracts/ 目录
- 提供方 CI 引入所有 consumer 的契约进行验证
```

### 1.3 集成测试策略

| 测试层级 | 使用 Mock | 使用真实调用 | 规则编号 |
|----------|-----------|-------------|----------|
| 单元测试 | 是 | 否 | TEST-INT-001 |
| 服务间集成测试 | 部分依赖 Mock | 数据库用 Testcontainers | TEST-INT-002 |
| 端到端测试 | 否 | 全链路真实调用 | TEST-INT-003 |

MUST：数据库集成测试使用 Testcontainers。

SHOULD：外部服务（支付网关、短信）使用 WireMock Mock。

### 1.4 端到端业务流程测试用例

| # | 业务流程 | 测试步骤 | 预期结果 | 规则编号 |
|---|----------|----------|----------|----------|
| 1 | 用户注册→登录 | 注册→激活→登录→获取 Token | Token 有效，用户信息正确 | TEST-E2E-001 |
| 2 | 商品浏览→下单→支付 | 浏览商品→加购物车→下单→支付 | 订单状态=已支付 | TEST-E2E-002 |
| 3 | 完整购物流程 | 下单→支付→发货→收货→评价 | 订单完成，评价可见 | TEST-E2E-003 |
| 4 | 退款流程 | 申请退款→审核→退款到账 | 退款状态=已退款 | TEST-E2E-004 |
| 5 | 秒杀流程 | 预热→抢购→下单→支付 | 库存扣减正确，无超卖 | TEST-E2E-005 |

## 第二部分：生产环境验证规范

### 2.1 生产冒烟测试用例集

MUST：每次部署后自动执行以下核心接口冒烟测试。

| # | 接口 | 验证内容 | 通过标准 | 规则编号 |
|---|------|----------|----------|----------|
| 1 | GET /actuator/health | 服务健康 | HTTP 200 + status=UP | TEST-SMK-001 |
| 2 | POST /api/auth/login | 用户登录 | HTTP 200 + Token 返回 | TEST-SMK-002 |
| 3 | GET /api/products | 商品列表 | HTTP 200 + 非空列表 | TEST-SMK-003 |
| 4 | GET /api/orders/{id} | 订单查询 | HTTP 200 + 订单数据 | TEST-SMK-004 |
| 5 | GET /api/user/profile | 用户信息 | HTTP 200 + 用户数据 | TEST-SMK-005 |

### 2.2 灰度验证用例

| 灰度阶段 | 验证逻辑 | 通过标准 | 规则编号 |
|----------|----------|----------|----------|
| 金丝雀（1%流量） | 核心接口无 5xx | 错误率 < 0.1% | TEST-GRAY-001 |
| 小流量（10%流量） | 核心流程无阻断 | 订单成功率 > 99% | TEST-GRAY-002 |
| 半量（50%流量） | 性能无退化 | P99 延迟 < 旧版本 1.2x | TEST-GRAY-003 |
| 全量（100%） | 全功能验证 | 所有冒烟测试通过 | TEST-GRAY-004 |

### 2.3 发布后监控观察清单

| 观察指标 | 观察时间 | 正常范围 | 异常处理 | 规则编号 |
|----------|----------|----------|----------|----------|
| HTTP 5xx 错误率 | 30 分钟 | < 0.1% | > 1% → 自动回滚 | TEST-OBS-001 |
| P99 响应延迟 | 30 分钟 | < 500ms | > 1000ms → 告警 | TEST-OBS-002 |
| JVM 内存使用 | 1 小时 | < 85% | > 90% → 告警 | TEST-OBS-003 |
| 数据库慢查询 | 1 小时 | < 5/min | > 20/min → 告警 | TEST-OBS-004 |
| 业务成功率 | 30 分钟 | > 99% | < 98% → 告警 | TEST-OBS-005 |

### 2.4 自动回滚触发条件

MUST：以下条件任一满足时自动回滚。

| 条件 | 阈值 | 触发方式 | 规则编号 |
|------|------|----------|----------|
| 错误率 | > 5% 持续 2 分钟 | Argo Rollouts / 手动 | TEST-ROLL-001 |
| 延迟 | P99 > 3000ms 持续 5 分钟 | 手动 | TEST-ROLL-002 |
| OOM Kill | 任何 Pod 发生 OOM | HPA + 告警 | TEST-ROLL-003 |
| 核心流程阻断 | 冒烟测试失败 | CI 自动 | TEST-ROLL-004 |

## 第三部分：测试数据管理规范

### 3.1 测试环境数据来源

| 数据来源 | 优先级 | 适用场景 | 规则编号 |
|----------|--------|----------|----------|
| Mock 数据 | 最高 | 单元测试、接口测试 | TEST-DATA-001 |
| 数据工厂生成 | 高 | 集成测试 | TEST-DATA-002 |
| 脱敏生产数据 | 中 | 性能测试、兼容性测试 | TEST-DATA-003 |
| 原始生产数据 | 禁止 | 任何环境 | TEST-DATA-004 |

### 3.2 生产数据脱敏流程

MUST：生产数据导入测试环境必须脱敏。

```
脱敏流程 SOP：
1. 提交脱敏申请（说明用途、所需字段、使用期限）
2. DBA 审批（评估数据量和敏感度）
3. 执行脱敏脚本（按脱敏规则表处理）
4. 安全员审核脱敏结果（抽样验证）
5. 导入测试环境
6. 使用完毕后清理（超过保留期限自动删除）
```

### 3.3 测试数据清理策略

| 数据类型 | 测试环境保留 | 生产环境 | 规则编号 |
|----------|-------------|----------|----------|
| PII 数据（手机号、身份证） | 禁止持久化 | 禁止出现在测试环境 | TEST-DATA-005 |
| 测试用户账号 | 测试期间有效 | N/A | TEST-DATA-006 |
| 自动生成的测试数据 | 每次测试后清理 | N/A | TEST-DATA-007 |
| 性能测试基准数据 | 版本发布前更新 | N/A | TEST-DATA-008 |

### 3.4 数据工厂模式

MUST：统一使用数据工厂创建测试数据。

```java
// 测试数据工厂
public class TestDataFactory {
    // 创建测试用户
    public static User createTestUser() {
        return User.builder()
            .username("test_" + UUID.randomUUID())
            .phone("138" + RandomUtil.randomNumbers(8))
            .email("test_" + UUID.randomUUID() + "@test.com")
            .status(UserStatus.ACTIVE)
            .build();
    }

    // 创建测试商品
    public static Product createTestProduct() {
        return Product.builder()
            .name("测试商品_" + UUID.randomUUID())
            .price(new BigDecimal("99.00"))
            .stock(100)
            .status(ProductStatus.ON_SALE)
            .build();
    }

    // 创建测试订单
    public static Order createTestOrder(Long userId, Long productId) {
        return Order.builder()
            .orderNo(OrderNoGenerator.generate())
            .userId(userId)
            .productId(productId)
            .status(OrderStatus.PENDING)
            .build();
    }
}
```

## 第四部分：非功能性回归测试规范

### 4.1 性能基线测试

MUST：每次发布前运行基准测试，对比上一版本。

| 测试场景 | 指标 | 基线 | 退化阈值 | 规则编号 |
|----------|------|------|----------|----------|
| 商品列表（100 并发） | P99 延迟 | < 200ms | > 240ms (+20%) | TEST-PERF-001 |
| 下单接口（50 并发） | P99 延迟 | < 500ms | > 600ms (+20%) | TEST-PERF-002 |
| 支付接口（20 并发） | P99 延迟 | < 300ms | > 360ms (+20%) | TEST-PERF-003 |
| 搜索接口（100 并发） | P99 延迟 | < 300ms | > 360ms (+20%) | TEST-PERF-004 |
| 商品详情（200 并发） | QPS | > 500 | < 400 (-20%) | TEST-PERF-005 |

### 4.2 性能退化自动阻断

MUST：P99 延迟增加 > 20% 时阻断发布。

```
阻断规则：
1. 发布前跑 JMeter/Gatling 基准测试
2. 对比上一版本的基准结果
3. 如 P99 延迟增加 > 20% → 发布阻断
4. 团队分析原因 → 优化 → 重新测试 → 通过后发布
```

### 4.3 安全回归测试

SHOULD：每次发布前跑 DAST 扫描。

| 扫描项 | 频率 | 工具 | 规则编号 |
|--------|------|------|----------|
| OWASP Top 10 | 每次发布 | OWASP ZAP | TEST-SEC-001 |
| API 安全测试 | 每次发布 | REST-Assured + 安全断言 | TEST-SEC-002 |
| 依赖漏洞 | 每次构建 | OWASP Dependency-Check | TEST-SEC-003 |

### 4.4 可观测性回归

SHOULD：每次发布后验证可观测性基础设施正常。

| 验证项 | 验证方法 | 规则编号 |
|--------|----------|----------|
| 日志正常输出 | 检查 ELK 日志是否有新错误 | TEST-OBS-006 |
| 指标正常采集 | 检查 Prometheus 是否有新指标 | TEST-OBS-007 |
| 链路追踪正常 | 检查 Zipkin 是否有完整调用链 | TEST-OBS-008 |
| 告警规则有效 | 检查 Alertmanager 规则是否触发 | TEST-OBS-009 |

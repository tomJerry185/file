0. 基本原则 | General Principles
================================================================================
0.1 规范分级说明 | Specification Levels
- MUST（必须）：强制要求，违反将导致安全/稳定性风险
- SHOULD（应该）：强烈推荐，例外需审批和文档记录
- MAY（可以）：可选建议，根据场景自行决定
- MUST NOT（禁止）：严格禁止，违反将触发违规处理

0.2 适用范围 | Scope
- 适用于本项目内所有服务端组件与基础设施对中间件的接入、配置、治理与联动使用
- 适用于开发/测试/预发/生产等所有环境

0.3 例外与豁免机制 | Exceptions & Waivers
- 例外必须经Owner/架构委员会审批
- 例外必须包含：原因/范围/风险/有效期/回滚方案/追踪编号
- 例外到期必须复审或清理
- 禁止无记录的口头例外
- 验证方式：例外审批记录/RFC/ADR

0.4 违规处理 | Violation Handling
- MUST级别违规：立即修复，并提交RCA
- SHOULD级别违规：5个工作日内修复或申请例外
- MAY级别违规：建议采纳，无强制要求
- 验证方式：违规跟踪系统/代码审查

0.5 术语表 | Glossary

> **通用术语**（代码审查、回滚、SLA/SLO、熔断、降级、幂等等）统一定义在《工程治理规范.md》第19.7节"术语表（完整版）"，此处仅列出中间件领域专有术语。

| 中文术语 | 英文术语 | 定义 |
|----------|----------|------|
| 热点Key | Hot Key | 访问频率远高于平均的缓存键 |
| 雪崩 | Cache Avalanche | 大量缓存同时失效导致的数据库压力激增 |
| 击穿 | Cache Penetration | 查询不存在的数据导致的数据库压力 |
| 污染 | Cache Pollution | 无效数据占用缓存空间导致的性能下降 |
| 追踪ID | TraceId | 分布式链路追踪的唯一标识符 |

2. 通用接入规范（MUST）| General Access Standards
================================================================================
2.1 配置管理 | Configuration Management
MUST 2.1.01 中间件地址/账号/密钥来源
- 所有敏感配置必须来自配置中心或环境变量
- 禁止在代码中硬编码任何连接信息/密钥/Token
- 生产环境配置不得使用默认值
- 验证方式：Secrets扫描/配置审查

```yaml
# 正确示例 ✓
spring:
  datasource:
    url: ${DB_URL}           # 来自环境变量
    username: ${DB_USER}
    password: ${DB_PASSWORD}

# 错误示例 ✗
spring:
  datasource:
    url: jdbc:mysql://192.168.1.100:3306/shop  # 硬编码
    username: root
    password: Root@123456
```

MUST 2.1.02 配置变更审计
- 所有配置变更必须记录：变更人/变更时间/变更内容/变更原因
- 配置必须支持回滚到历史版本
- 配置变更必须通知相关团队
- 验证方式：审计日志/配置中心记录

2.2 连接与超时配置 | Connection & Timeout Configuration
MUST 2.2.01 超时配置要求
- 所有中间件调用必须配置超时时间
- 超时时间必须根据业务特性设定，不得使用无限超时

| 中间件类型 | 连接超时 | 读取超时 | 说明 |
|-----------|----------|----------|------|
| HTTP/Feign| 3s       | 30s      | 可按接口调整 |
| Redis     | 2s       | 5s       | 简单操作 |
| MySQL     | 5s       | 30s      | 复杂查询可更长 |
| RocketMQ  | 3s       | -        | 发送超时 |
| ES        | 5s       | 60s      | 复杂聚合查询 |

- 验证方式：配置审查/集成测试

MUST 2.2.02 重试策略
- 必须配置重试上限，禁止无限重试
- 必须使用退避策略（指数退避/固定延迟）
- 只对幂等操作启用重试
- 验证方式：配置审查

```yaml
# 正确示例 ✓
spring:
  cloud:
    openfeign:
      client:
        config:
          default:
            connectTimeout: 3000
            readTimeout: 30000
            maxAttempts: 3               # 最多重试3次
            backoff:
              enabled: true
              multiplier: 2              # 指数退避
              initialInterval: 100ms     # 初始间隔

# 错误示例 ✗
maxAttempts: -1  # 无限重试
```

MUST 2.2.03 连接池配置
- 必须配置连接池上限、空闲回收、健康检查
- 连接池大小必须根据容量评估设定

| 中间件 | 最小连接 | 最大连接 | 空闲回收 | 健康检查 |
|--------|----------|----------|----------|----------|
| MySQL  | 5        | 100      | 10min    | enabled  |
| Redis  | 5        | 50       | 5min     | enabled  |
| HTTP   | 5        | 200      | 60s      | -        |

- 验证方式：性能测试/配置审查

2.3 可观测性 | Observability
MUST 2.3.01 日志要求
- 所有关键操作必须记录日志
- 日志必须包含：traceId/业务主键/操作结果
- 日志级别：ERROR（错误）/WARN（警告）/INFO（关键操作）
- 验证方式：日志审查/链路追踪

```java
// 正确示例 ✓
log.info("Order created, orderId={}, traceId={}", order.getId(), TraceContext.getTraceId());
log.error("Payment failed, orderId={}, error={}", orderId, e.getMessage(), e);

// 错误示例 ✗
log.info("Order created");  // 缺少关键信息
log.info("Payment failed: " + e);  // 异常未正确处理
```

MUST 2.3.02 指标要求
- 必须暴露核心指标：QPS/延迟/错误率/资源使用率
- 指标命名规范：`{domain}_{subdomain}_{metric}_{unit}`

| 指标类型 | 命名示例 | 说明 |
|----------|----------|------|
| 计数器   | api_requests_total | 总请求数 |
| 仪表盘   | connection_pool_active | 活动连接数 |
| 直方图   | api_duration_seconds | 请求延迟分布 |
| 摘要     | cache_hit_ratio | 缓存命中率 |

- 验证方式：Prometheus配置/监控Dashboard

MUST 2.3.03 链路追踪
- traceId必须贯穿全链路
- 所有异步任务必须传递traceId
- 采样率可配置：生产环境1-10%，开发环境100%
- 验证方式：Zipkin/Jaeger验证

2.4 环境隔离 | Environment Isolation
MUST 2.4.01 环境隔离要求
- 开发/测试/预发/生产必须使用独立的中间件实例
- 禁止跨环境复用同一实例
- 环境命名规范：dev/test/staging/prod
- 验证方式：架构审查/配置扫描

MUST 2.4.02 数据隔离
- 敏感数据必须在生产环境加密存储
- 测试数据不得包含真实PII信息
- 验证方式：数据审计/安全扫描

2.5 安全与合规 | Security & Compliance
MUST 2.5.01 认证与授权
- 生产环境必须启用认证授权
- 必须使用最小权限原则
- 禁止使用默认账号/弱密码
- 验证方式：安全扫描/渗透测试

MUST 2.5.02 传输加密
- 生产环境必须启用TLS/SSL
- 禁止使用已弃用的TLS版本（v1.0/v1.1）
- 验证方式：SSL Labs测试/配置审查

```yaml
# 正确示例 ✓
spring:
  redis:
    ssl: true
    protocol: TLSv1.2,TLSv1.3

# 错误示例 ✗
spring:
  redis:
    ssl: false  # 生产环境禁用SSL
```

MUST 2.5.03 敏感数据处理
- 日志/指标禁止包含PII明文
- 密码/Token必须脱敏后记录
- 验证方式：日志扫描/安全审查

2.6 容错与降级 | Fault Tolerance & Degradation
MUST 2.6.01 降级预案
- 所有外部依赖必须具备降级方案
- 降级开关必须可动态配置
- 验证方式：架构审查/降级演练

MUST 2.6.02 熔断配置
- 必须配置熔断阈值和恢复策略
- 熔断后不得返回伪成功
- 验证方式：熔断测试/代码审查

```java
// 正确示例 ✓
@CircuitBreaker(name = "userService", fallbackMethod = "getUserFallback")
public User getUser(Long id) {
    return userClient.getUser(id);
}

public User getUserFallback(Long id, Exception e) {
    log.warn("UserService unavailable, using cache, id={}", id);
    return userCache.get(id);  // 明确的降级逻辑
}

// 错误示例 ✗
@CircuitBreaker(name = "userService")
public User getUser(Long id) {
    try {
        return userClient.getUser(id);
    } catch (Exception e) {
        return new User();  // 空对象，伪成功
    }
}
```

MUST 2.6.03 回滚要求
- 所有发布必须可回滚
- 回滚时间目标 <= 10分钟
- 必须定期演练回滚流程
- 验证方式：回滚演练记录

3. 中间件搭配使用规则（MUST）| Integration Rules
================================================================================
3.1 缓存与数据库 | Cache & Database
MUST 3.1.01 缓存一致性策略
- 写路径必须明确采用：Cache-Aside/Write-Through/Write-Behind
- 本项目统一使用：Cache-Aside（旁路缓存）
- 验证方式：代码审查

```java
// Cache-Aside 模式 ✓
public Product getProduct(Long id) {
    // 1. 先查缓存
    Product product = cache.get(id);
    if (product != null) {
        return product;
    }
    // 2. 缓存未命中，查数据库
    product = productMapper.selectById(id);
    if (product != null) {
        // 3. 写入缓存
        cache.set(id, product, TTL);
    }
    return product;
}

public void updateProduct(Product product) {
    // 1. 先更新数据库
    productMapper.updateById(product);
    // 2. 删除缓存（而非更新）
    cache.delete(product.getId());
}
```

MUST 3.1.02 缓存失效策略
- 业务缓存必须设置TTL，禁止永久缓存
- 热点Key必须保护：互斥锁/提前过期/本地缓存
- 验证方式：代码审查/缓存分析

| 缓存类型 | 默认TTL | 说明 |
|----------|---------|------|
| 商品信息 | 30min   | 可接受短暂延迟 |
| 用户信息 | 15min   | 变更频率中等 |
| 配置信息 | 5min    | 需快速生效 |
| 热点商品 | 1min    + 互斥锁 | 防击穿 |
| 计数器   | 不过期  + 定期持久化 | 防丢失 |

MUST 3.1.03 缓存防护
- 缓存雪崩防护：TTL增加随机偏移（±10%）
- 缓存击穿防护：热点Key使用互斥锁
- 缓存污染防护：设置最大内存+淘汰策略
- 验证方式：代码审查/压测

3.2 消息队列与数据库 | MQ & Database
MUST 3.2.01 幂等性要求
- 消费端必须支持重复消息处理
- 必须使用业务唯一键做幂等控制
- 验证方式：幂等测试/代码审查

```java
// 正确示例 ✓
@RocketMQMessageListener(topic = "order-topic", consumerGroup = "order-group")
public class OrderConsumer implements RocketMQListener<OrderMessage> {
    @Override
    public void onMessage(OrderMessage message) {
        // 使用唯一键保证幂等
        String uniqueKey = "order:" + message.getOrderId();
        if (redis.setnx(uniqueKey, "1", 24h) == 0) {
            log.info("Duplicate message, orderId={}", message.getOrderId());
            return;  // 重复消息，直接返回
        }
        // 处理业务逻辑
        processOrder(message);
    }
}
```

MUST 3.2.02 事务消息
- 事务与消息必须采用可靠投递策略
- 推荐方案：本地消息表/Outbox模式/事务消息
- 禁止：DB事务 + MQ发送嵌套在同一事务中
- 验证方式：架构审查/混沌测试

```
正确方案 ✓：本地消息表
BEGIN TRANSACTION
1. 业务操作（INSERT/UPDATE）
2. 插入消息表（status=PENDING）
COMMIT

异步任务：
3. 扫描消息表
4. 发送MQ
5. 更新状态（SENT/FAILED）

错误方案 ✗：
BEGIN TRANSACTION
1. 业务操作
2. 发送MQ（可能失败/重复）
COMMIT  -- MQ发送不在事务中
```

MUST 3.2.03 死信队列处理
- 必须配置死信队列
- 死信消息必须告警
- 必须有死信消息重放机制
- 验证方式：MQ配置审查/告警测试

3.3 网关与服务 | Gateway & Service
MUST 3.3.01 限流统一策略
- 网关限流与服务限流必须协同
- 网关限流 > 服务限流，避免放大失败
- 限流算法推荐：令牌桶/滑动窗口
- 验证方式：架构审查/压测

```
正确配置 ✓：
网关限流：1000 QPS
服务限流：800 QPS  （留有余量）

错误配置 ✗：
网关限流：1000 QPS
服务限流：1200 QPS  （服务限流无效）
```

MUST 3.3.02 配置中心化
- CORS/熔断/限流配置必须存储在Nacos
- 配置变更必须支持动态刷新，无需重启
- 验证方式：配置审查/动态刷新测试

3.4 分布式事务 | Distributed Transaction
MUST 3.4.01 Seata使用规范
- 必须明确全局事务边界
- 禁止将长耗时逻辑放入全局事务
- 全局事务超时 <= 60秒
- 验证方式：代码审查/事务测试

```java
// 正确示例 ✓
@GlobalTransactional(timeoutMills = 60000)
public void createOrder(OrderDTO dto) {
    // 仅包含核心业务逻辑
    orderService.create(dto);
    inventoryService.deduct(dto.getProductId(), dto.getCount());
    // 不包含：RPC调用/外部请求/耗时计算
}

// 错误示例 ✗
@GlobalTransactional
public void createOrder(OrderDTO dto) {
    orderService.create(dto);
    inventoryService.deduct(dto.getProductId(), dto.getCount());
    paymentService.callThirdParty(dto);  // 外部RPC，违反规则
    notificationService.sendEmail(dto);  // 非核心逻辑
}
```

MUST 3.4.02 分布式事务与消息队列
- 分布式事务与消息队列不得嵌套
- 必须使用补偿/最终一致性方案
- 验证方式：架构审查

# 正确配置 ✓
logging:

4.2 Spring Cloud Gateway | API Gateway
MUST 4.2.01 超时与连接配置
```yaml
# 推荐配置
spring:
  cloud:
    gateway:
      httpclient:
        connect-timeout: 3000          # 连接超时3秒
        response-timeout: 60s          # 响应超时60秒
        pool:
          max-connections: 500         # 最大连接数
          max-idle-time: 20s           # 空闲回收时间
          max-life-time: 60s           # 连接最大存活时间
```

// 正确示例 ✓

    // 等待时间3秒，锁自动释放时间10秒
    boolean acquired = lock.tryLock(3, 10, TimeUnit.SECONDS);

// TCC模式：适合复杂业务
@LocalTCC
public class StockTccService {
    @TwoPhaseBusinessAction(name = "stockDeduct", commitMethod = "commit", rollbackMethod = "rollback")
    public boolean prepare(BusinessActionContext context, int count) {
        // 预留资源
        return stockDao.deduct(count, 0);  // 冻结库存
    }

    public boolean commit(BusinessActionContext context) {
        // 确认扣减
        return stockDao.deduct(count, 1);
    }

    public boolean rollback(BusinessActionContext context) {
        // 释放冻结
        return stockDao.release(count);
    }
}

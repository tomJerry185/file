## 第一部分：事故→规则闭环流程（Task 5.4）

### 1.1 RCA 模板增强

MUST：RCA 报告必须包含"规范改进建议"必填字段。

RCA 报告新增字段：

```
## 规范改进建议（必填）

### 发现的规范缺口
- [ ] 现有规范是否覆盖了此类问题？是/否
- [ ] 如果否，建议新增/修改哪条规范？

### 具体改进建议
| # | 建议内容 | 影响的规范文件 | 优先级 | 负责人 | 截止日期 |
|---|----------|---------------|--------|--------|----------|
| 1 |         |               |        |        |          |

### 自动化检查建议
- [ ] 是否需要新增自动化检查规则？
- [ ] 检查脚本名称：
- [ ] 检查规则编号：
```

### 1.2 月度审查流程

```
每月第一个周一，架构委员会会议：
1. 审查上月所有 RCA 报告的规范改进建议
2. 评估每条建议的优先级
3. 确认后分配给对应团队
4. 创建规范变更 PR
5. 同步更新自动化检查脚本
6. 全量扫描存量代码中的同类问题
```

### 1.3 度量

| 指标 | 目标 | 计算方式 |
|------|------|----------|
| 事故→规则转化率 | > 80% | 产出新规则的 RCA 数 / 总 RCA 数 |
| 新规则自动化率 | > 60% | 有自动化检查的新规则数 / 新规则总数 |
| 存量问题修复率 | > 50% | 已修复同类问题数 / 扫描发现总数 |

## 第三部分：新人 Onboarding 路径（Task 5.6）

### 3.1 Day 1：规范体系概览

**时间：2 小时**

1. 阅读 `rule/快速参考-后端.md`（30 分钟）
2. 阅读 `rule/快速参考-前端.md`（30 分钟）
3. 阅读 `rule/快速参考-中间件.md`（20 分钟）
4. 阅读 `rule/SKILL.md` 了解 AI 辅助规则（20 分钟）
5. 安装开发环境：运行 `scripts/setup-dev-env.sh`（20 分钟）

**验证**：能说出规范体系的 5 个文件和各自的作用。

### 3.2 Day 2-3：角色对应的深度阅读

**后端研发**：
1. `rule/后端规范.md` — 重点阅读第 2、5、6、10 章
2. `rule/安全防护规范.md` — SQL 注入防护、XSS 防护
3. `rule/数据安全与隐私规范.md` — 数据分级、脱敏规则
4. `rule/测试完备性规范.md` — 测试策略

**前端研发**：
1. `rule/前端规范.md` — 重点阅读第 1、2、3、9 章
2. `rule/安全防护规范.md` — XSS 防护、CSP、v-html
3. `rule/测试完备性规范.md` — 前端测试策略

**运维/SRE**：
1. `rule/中间件规范.md` — 重点阅读第 2、4 章
2. `rule/运维与可靠性规范.md` — Runbook、容量、灾备
3. `rule/安全事件应急规范.md` — 应急响应 SOP

**验证**：完成对应角色的阅读检查清单（导师签字）。

### 3.3 Day 4：合规 PR 实践

1. 在导师指导下完成一个合规 PR
2. PR 必须通过 CI 规范检查
3. 导师 Code Review 检查规范遵守情况
4. 修复至少 1 个存量违规（从 suppression 列表中选择）

**验证**：PR 通过 CI + Code Review。

### 3.4 Day 5：规范知识测试（可选）

20 道选择题，涵盖：
- MUST 规则识别（8 题）
- 安全规范（6 题）
- 运维规范（3 题）
- 测试规范（3 题）

**通过标准**：≥ 80 分。

## 第五部分：规范定期复审自动化（Task 5.8）

### 5.1 GitHub Actions 定期检查

MUST：自动检查规范文件的"下次复审"字段，过期自动创建 Issue。

```yaml
# .github/workflows/rule-review-check.yml（建议创建）
name: Rule Review Check
on:
  schedule:
    - cron: '0 9 1 * *'  # 每月1日 9:00 检查
  workflow_dispatch:

jobs:
  check-review-dates:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Check rule review dates
        run: |
          EXPIRED_RULES=""
          for file in rule/*.md; do
            # 提取"下次复审"日期
            review_date=$(grep "下次复审" "$file" | grep -oP '\d{4}-\d{2}-\d{2}' || echo "")
            if [ -n "$review_date" ]; then
              review_epoch=$(date -d "$review_date" +%s 2>/dev/null || echo "0")
              today_epoch=$(date +%s)
              if [ "$review_epoch" -lt "$today_epoch" ]; then
                EXPIRED_RULES="$EXPIRED_RULES\n- $file (复审日期: $review_date)"
              fi
            fi
          done

          if [ -n "$EXPIRED_RULES" ]; then
            echo -e "以下规范文件已过复审日期：\n$EXPIRED_RULES"
            echo "EXPIRED=true" >> $GITHUB_ENV
            echo "EXPIRED_RULES=$EXPIRED_RULES" >> $GITHUB_ENV
          else
            echo "所有规范文件均在复审有效期内。"
            echo "EXPIRED=false" >> $GITHUB_ENV
          fi

      - name: Create review issue if expired
        if: env.EXPIRED == 'true'
        uses: actions/github-script@v7
        with:
          script: |
            const expiredRules = process.env.EXPIRED_RULES;
            await github.rest.issues.create({
              owner: context.repo.owner,
              repo: context.repo.repo,
              title: `规范复审提醒 — ${new Date().toISOString().split('T')[0]}`,
              body: `以下规范文件的"下次复审"日期已过期，请安排复审：\n${expiredRules}\n\n---\n\n复核清单：\n- [ ] 检查规则是否仍然适用\n- [ ] 更新版本号和复审日期\n- [ ] 记录变更到变更历史\n- [ ] 更新 MEMORY.md（如有结构性变更）`,
              labels: ['rule-review', 'documentation']
            });
```

### 5.2 复审结果记录

MUST：复审结果记录到变更历史。

```
复审记录模板：
- 规则是否仍然适用？
- 是否需要新增/删除/修改规则？
- 行业标准是否有更新？
- 实际执行中遇到了什么问题？
- 下次复审日期：
```

## 1. 安全事件分级

> MUST：安全事件必须按以下标准分级，不同级别对应不同响应时效。

| 级别 | 定义 | 数据泄露量 | 影响范围 | 响应时效 | 规则编号 |
|------|------|-----------|----------|----------|----------|
| S0 - 紧急 | 大规模数据泄露、系统被完全控制 | > 10万条 PII | 全部用户 | 15 分钟内响应 | SEC-IR-001 |
| S1 - 严重 | 敏感数据泄露、支付异常、勒索攻击 | 1千~10万条 PII | 大量用户 | 30 分钟内响应 | SEC-IR-002 |
| S2 - 高危 | 局部数据泄露、系统被渗透、SQL 注入成功 | < 1千条 PII | 部分用户 | 2 小时内响应 | SEC-IR-003 |
| S3 - 中危 | 异常访问、可疑活动、配置错误 | 无直接泄露 | 少量用户 | 8 小时内响应 | SEC-IR-004 |
| S4 - 低危 | 扫描探测、日志异常、弱密码 | 无泄露 | 无 | 24 小时内响应 | SEC-IR-005 |

### 1.1 事件类型对照

| 事件类型 | 默认级别 | 升级条件 |
|----------|----------|----------|
| 数据泄露（PII） | S1 | 泄露量 > 10万 → S0 |
| 支付系统异常 | S1 | 资金损失 > 10万 → S0 |
| 勒索软件/勒索攻击 | S1 | 核心业务受影响 → S0 |
| SQL 注入（成功） | S2 | 获取到数据 → S1 |
| XSS（成功利用） | S2 | 影响管理员 → S1 |
| DDoS 攻击 | S2 | 服务完全不可用 → S1 |
| 暴力破解 | S3 | 成功登录 → S2 |
| 异常访问模式 | S3 | 内部人员 → S2 |
| 端口扫描/探测 | S4 | 持续性探测 → S3 |
| 合规检查不通过 | S3 | 高风险项 → S2 |

## 2. 应急响应 SOP

> MUST：安全事件处理遵循以下标准操作流程。

### 2.1 响应流程总览

```text
发现 → 评估 → 遏制 → 根除 → 恢复 → 复盘
 │      │      │      │      │      │
 ▼      ▼      ▼      ▼      ▼      ▼
监测   定级   止损   修复   验证   RCA
告警   通知   隔离   加固   恢复   改进
```

### 2.2 S0/S1 事件详细 SOP

**T+0 ~ T+15min：发现与评估**

1. 接收告警（监控系统/人工发现/外部通报）
2. 值班人员确认事件真实性
3. 初步定级（S0/S1/S2/S3/S4）
4. 通知应急响应团队

**T+15min ~ T+30min：遏制（止损）**

5. 隔离受影响系统（网络隔离/服务下线）
6. 切断攻击路径（封禁 IP/撤销凭证）
7. 保存现场（快照/日志备份）
8. 启动应急通信渠道

**T+30min ~ T+4h：根除**

9. 查找攻击入口和影响范围
10. 清除恶意代码/关闭后门
11. 重置受影响账号密码
12. 修补安全漏洞

**T+4h ~ T+24h：恢复**

13. 从干净备份恢复数据（如需要）
14. 逐步恢复服务（先内部验证，再灰度上线）
15. 加强监控（针对同类攻击增强告警）
16. 验证业务功能正常

**T+24h ~ T+7d：复盘**

17. 编写 RCA 报告
18. 提出改进措施
19. 跟进改进项落地

### 2.3 通信渠道

| 级别 | 内部通信 | 外部通信 |
|------|----------|----------|
| S0 | 电话 + 钉钉紧急群 + 现场集结 | 法务+公关+监管（24h内） |
| S1 | 钉钉紧急群 + 现场负责人 | 法务+公关（48h内评估） |
| S2 | 钉钉安全群 | 视情况决定 |
| S3 | 钉钉安全群 + 邮件 | 不需要 |
| S4 | 邮件 | 不需要 |

## 3. 通知机制

> MUST：S0/S1 事件必须在规定时限内通知相关方。

### 3.1 通知时限表

| 通知对象 | S0 时限 | S1 时限 | S2 时限 |
|----------|---------|---------|---------|
| CTO/CEO | 30 分钟 | 2 小时 | 24 小时 |
| 安全团队全员 | 15 分钟 | 30 分钟 | 4 小时 |
| 法务团队 | 4 小时 | 24 小时 | 72 小时 |
| 公关团队 | 4 小时 | 48 小时 | 不需要 |
| 受影响用户 | 72 小时 | 7 天 | 不需要 |
| 监管机构 | 按法规要求 | 按法规要求 | 不需要 |
| 执法机关 | 按法规要求 | 按法规要求 | 不需要 |

### 3.2 用户通知模板

```text
【安全通知】尊敬的用户：

我们于 [日期] 发现一起安全事件，可能影响了您的 [数据类型]。
事件详情：
- 发生时间：[时间]
- 影响范围：[描述]
- 我们已采取的措施：[措施]
- 您应采取的措施：[建议]

如您有任何疑问，请联系客服：[联系方式]。

我们对此深表歉意，并正在全力加强安全防护。

[公司名称]
[日期]
```

### 3.3 监管通知

MUST：涉及个人信息泄露的事件，按《个人信息保护法》要求通知。

- 泄露个人信息：72 小时内通知网信部门
- 泄露金融信息：立即通知银保监/人行
- 涉及跨境数据：同时通知数据出境相关监管

## 4. 漏洞披露流程

> MUST：对外安全漏洞披露遵循负责任披露原则。

### 4.1 内部发现漏洞

```text
1. 发现漏洞 → 记录到漏洞管理系统
2. 评估严重程度（CVSS 评分）
3. 制定修复计划
4. 修复并测试
5. 部署修复
6. 更新安全公告
```

### 4.2 外部报告漏洞

```text
1. 接收外部漏洞报告（security@example.com）
2. 24 小时内确认收到并初步评估
3. 7 天内反馈处理计划
4. 修复后通知报告者
5. 按约定时间公开披露（默认 90 天）
6. 致谢安全研究者
```

### 4.3 披露时间线

| 漏洞级别 | 内部修复时限 | 公开披露时限 |
|----------|-------------|-------------|
| Critical (CVSS ≥ 9.0) | 24 小时 | 修复后 7 天 |
| High (CVSS 7.0-8.9) | 7 天 | 修复后 30 天 |
| Medium (CVSS 4.0-6.9) | 30 天 | 修复后 90 天 |
| Low (CVSS < 4.0) | 90 天 | 修复后 90 天 |

## 5. 取证与保留规范

> MUST：安全事件相关的证据必须完整保留，禁止提前清理。

### 5.1 证据类型与保留要求

| 证据类型 | 保留期限 | 收集方式 |
|----------|----------|----------|
| 服务器快照 | 事件解决后 1 年 | 云平台快照 |
| 网络流量日志 | 事件解决后 1 年 | 流量镜像/NDR |
| 应用日志 | 事件解决后 1 年 | ELK 归档 |
| 数据库审计日志 | 永久 | 审计库 |
| 攻击样本（恶意文件） | 永久 | 隔离存储 |
| 内存转储 | 事件解决后 1 年 | 专用工具采集 |
| 沟通记录（钉钉/邮件） | 永久 | 原始存档 |

### 5.2 证据保全原则

1. **完整性**：使用哈希校验确保证据未被篡改
2. **不可变性**：证据写入后禁止修改（WORM 存储）
3. **可追溯性**：每个证据记录采集人、时间、地点
4. **保密性**：证据仅限应急团队和法律团队访问

### 5.3 禁止事项

MUST：事件调查期间禁止以下操作：
- 重启受影响服务器（除非必要止损）
- 清理日志文件
- 修改系统时间
- 运行修复脚本（除非确认不破坏证据）
- 向无关人员透露事件详情

## 6. 事后复盘（RCA）

> MUST：S0/S1 事件必须在 7 天内完成 RCA，S2 在 14 天内完成。

### 6.1 RCA 报告模板

```text
1. 事件摘要
   - 事件名称、级别、时间线

2. 影响评估
   - 影响用户数、数据泄露量、业务损失

3. 根因分析（5 Whys）
   - 技术根因
   - 流程根因
   - 人为根因

4. 时间线（精确到分钟）
   - 发现时间、响应时间、遏制时间、恢复时间

5. 改进措施
   - 技术改进（代码/配置/架构）
   - 流程改进（监控/告警/响应）
   - 规范改进（新增或加强规则）

6. 行动项追踪
   - 每个改进项：负责人、截止日期、优先级
```

### 6.2 改进措施闭环

MUST：RCA 结论必须转化为具体的改进行动。

```text
RCA 结论 → 行动项 → 跟踪 → 验证 → 关闭
   │                                    │
   └──→ 规范更新（新增/加强规则）────────┘
```

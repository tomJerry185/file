## 1. 术语表与核心概念
---

### 1.1 统一用语（UNIFIED_TERMINOLOGY）

| 中文术语 | 英文术语 | 缩写 | 定义 |
|----------|----------|------|------|
| **代码审查** | Code Review | CR | 对代码变更进行的系统性检查 |
| **审查清单** | Review Checklist | | 代码审查时使用的检查项 |
| **回滚** | Rollback | | 将系统恢复到之前版本的操作 |
| **需求分析** | Requirements Analysis | RA | 确定用户需求和系统约束的过程 |
| **用户故事** | User Story | US | 描述用户价值的简短需求描述 |
| **验收标准** | Acceptance Criteria | AC | 需求完成的验证条件 |
| **架构决策记录** | Architecture Decision Record | ADR | 架构决策的文档记录 |
| **根因分析** | Root Cause Analysis | RCA | 事故根本原因分析 |
| **隐私影响评估** | Privacy Impact Assessment | PIA | 隐私风险评估 |

### 1.2 核心概念（CORE_CONCEPTS）

```
【需求层次】
愿景 → 战略 → Epic → Feature → User Story → Task

【需求优先级（MoSCoW）】
- Must Have：必须有，没有则无法发布
- Should Have：应该有，高价值但可延后
- Could Have：可以有，有时间再做
- Won't Have：本次不做，明确排除

【敏捷术语】
- Sprint：2-4周的迭代周期
- Backlog：待办事项列表
- Kanban：看板，可视化工作流
- Daily Standup：每日站会

【DevOps术语】
- CI：Continuous Integration，持续集成
- CD：Continuous Deployment/Delivery，持续部署/交付
- Pipeline：流水线，自动化工作流
- Build：构建，编译打包过程
- Deploy：部署，发布到环境
- Rollback：回滚，撤销部署
```

### 2.3 AI提示词补充规范（AI_PROMPT_STANDARDS）

#### 2.3.1 任务边界与目标清晰

```
【要求】
- 提示词必须明确任务类型：生成/修改/修复/重构/测试/文档
- 单次提示仅执行一个动作
- 复杂需求必须拆分为多轮提示或多个子任务
```

#### 2.3.2 上下文与相关代码

```
【要求】
- 提示词必须提供：文件路径、函数/类名、业务约束、输入输出
- 修复场景需包含：复现步骤、期望行为、实际行为
- 提示词必须明确：版本、配置、运行环境
- 需要检索代码时，提供import/引用/符号名或文件路径
```

#### 2.3.3 具体、无歧义与输出格式

```
【要求】
- 禁止模糊词汇："优化一下""修一下""改一下"
- 必须量化目标：性能指标、边界条件、格式要求
- 明确输出格式：unified diff/修改后的函数/修改清单+验证步骤
- 指令结构一致：使用分隔符或标题分区（Context/Task/Constraints/Output）
```

#### 2.3.4 示例与测试驱动

```
【要求】
- 优先提供：示例输入/输出、失败用例
- 修复类请求必须提供：最小复现步骤
- 需要新增/更新测试时，要求：测试用例或测试计划
```

#### 2.3.5 迭代与核验

```
【要求】
- 要求AI明确假设与缺失信息
- 信息不足时必须先提问，不生成占位实现
- 要求输出验证步骤：测试命令/运行路径/回滚提示
```

#### 2.3.6 安全与合规约束

```
【要求】
- 禁止生成：敏感信息、密钥、后门代码
- 要求遵循：仓库规范、日志策略、权限策略
```

### 3.5 后端代码审查问题清单（BACKEND_ISSUES）

#### 3.5.1 禁止返回捏造的假数据（MUST）

**问题描述**：在服务不可用或业务失败时，返回硬编码的假数据伪造成功状态。

```java
// ✗ 错误示例
private Map<String, Object> performTokenRefresh(String refreshToken) {
    Map<String, Object> tokenResponse = new HashMap<>();
    tokenResponse.put("access_token", "new_access_token_" + System.currentTimeMillis());
    tokenResponse.put("expires_in", 3600);
    return tokenResponse;  // 假数据！
}

// ✓ 正确示例
private Map<String, Object> performTokenRefresh(String refreshToken) {
    try {
        String tokenEndpoint = properties.getTokenEndpoint();
        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create(tokenEndpoint))
            .POST(BodyPublishers.ofString(requestBody))
            .build();
        HttpResponse<String> response = client.send(request, BodyHandlers.ofString());

        if (response.statusCode() == 200) {
            return parseTokenResponse(response.body());
        }
        return null;
    } catch (Exception e) {
        log.error("令牌刷新失败", e);
        throw new RuntimeException("令牌刷新失败", e);
    }
}
```

**检查规则**：
- [ ] 禁止使用 `System.currentTimeMillis()` 生成假 ID/token
- [ ] 禁止返回硬编码字符串作为业务数据
- [ ] 降级方法必须返回错误或抛异常
- [ ] OAuth2/OpenID Connect 必须真实调用

#### 3.5.2 实体类完整性（MUST）

```java
// ✓ 使用 Lombok
@Data
@AllArgsConstructor
@NoArgsConstructor
public class Result<T> {
    private Boolean success;
    private String message;
    private T data;
}

// ✓ 手动编写
public class Result<T> {
    private Boolean success;
    private String message;
    private T data;

    public Boolean getSuccess() { return success; }
    public void setSuccess(Boolean success) { this.success = success; }
    // ... 其他 getter/setter
}
```

**检查规则**：
- [ ] 实体类必须有 Getter/Setter 或 @Data
- [ ] 数据库实体必须有 @TableName
- [ ] 乐观锁字段必须有 @Version

#### 3.5.3 降级处理规范（MUST）

```java
// ✗ 错误：降级时返回假数据
@Component
public class OrderFeignFallback implements OrderFeignClient {
    public OrderInfo getOrderByPaymentId(String paymentId) {
        OrderInfo info = new OrderInfo();
        info.setId(0L);
        info.setStatus("PENDING");
        return info;  // 伪装成功！
    }
}

// ✓ 正确：降级时抛出异常
@Component
public class OrderFeignFallback implements OrderFeignClient {
    public OrderInfo getOrderByPaymentId(String paymentId) {
        log.error("[Feign降级] 订单服务不可用，paymentId={}", paymentId);
        throw new PaymentException(
            PaymentErrorCode.ORDER_SERVICE_UNAVAILABLE,
            "订单服务暂时不可用，请稍后重试"
        );
    }
}
```

**检查规则**：
- [ ] Fallback 禁止返回 Result.success
- [ ] 查询降级禁止返回空结果伪装成功
- [ ] 必须有明确错误日志
- [ ] 错误信息说明服务不可用

## 4. 事件管理规范（ITIL）
---

### 4.1 事件生命周期（INCIDENT_LIFECYCLE）

```
┌─────────────────────────────────────────────────────────────┐
│                     事件生命周期                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  检测 → 分类 → 响应 → 解决 → 关闭                             │
│   ↓      ↓      ↓      ↓      ↓                             │
│ 告警   分级   处理   修复   复盘                             │
│                                                         │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 事件分级（INCIDENT_CLASSIFICATION）

| 级别 | 代码 | 响应SLA | 修复SLA | 示例 |
|------|------|---------|---------|------|
| **P0** | CRITICAL | 15分钟 | 2小时缓解 | 生产全局不可用 |
| **P1** | HIGH | 30分钟 | 4小时缓解 | 核心功能大面积受影响 |
| **P2** | MEDIUM | 2小时 | 24小时修复 | 局部功能受影响 |
| **P3** | LOW | 1工作日 | 3工作日 | 低影响/无用户感知 |

### 4.3 升级路径（ESCALATION_PATH）

```

## 5. 根因分析（RCA）规范

### 5.1 RCA方法论（RCA_METHODOLOGIES）

#### 5.1.1 5 Whys方法

```
【示例：数据库连接池耗尽】
问题：应用无法连接数据库

Why 1? 为什么无法连接？
答：连接池已耗尽

Why 2? 为什么连接池耗尽？
答：连接未释放

Why 3? 为什么连接未释放？
答：代码中有finally块未执行

Why 4? 为什么finally未执行？
答：某些异常导致JVM崩溃

Why 5? 为什么JVM会崩溃？
答：存在内存泄漏导致OOM

【根本原因】内存泄漏导致JVM崩溃
【解决方案】修复内存泄漏
```

### 11.2 Git Hooks配置（GIT_HOOKS）

```bash
# .git/hooks/pre-commit
#!/bin/bash

# 检查假数据关键词
if git diff --cached --name-only | grep "\.java$"; then
    if git diff --cached | grep "new_access_token_" > /dev/null; then
        echo "错误：代码中包含假数据关键词 'new_access_token_'"
        exit 1
    fi
    if git diff --cached | grep "oauth_user" > /dev/null; then
        echo "错误：代码中包含假数据关键词 'oauth_user'"
        exit 1
    fi
fi

# 检查文件结尾
if git diff --cached --name-only | grep "\.java$"; then
    if git diff --cached | grep -v "^+$" | tail -1 | grep -v "^$" | grep -v ".$"; then
        echo "警告：Java文件应以换行符结尾"
    fi
fi

exit 0
```

### 11.4 自动化脚本（AUTOMATION_SCRIPTS）

```bash
#!/bin/bash
# scripts/check-fake-data.sh

# 检查假数据关键词
echo "检查假数据关键词..."

PATTERNS=(
    "new_access_token_"
    "oauth_user"
    "fallback_access_token_"
    "test_user_.*_token"
)

for pattern in "${PATTERNS[@]}"; do
    if grep -r "$pattern" src/main/java; then
        echo "❌ 发现假数据关键词: $pattern"
        exit 1
    fi
done

echo "✅ 假数据检查通过"
```
